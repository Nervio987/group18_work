# 智汇办公 AI 智能平台 - 需求文档

## 1. 项目概述

### 1.1 项目名称
智汇办公 AI 智能平台（Zhihui Office AI Platform）

### 1.2 项目背景
随着企业知识资产的快速增长（技术文档、规章制度、行政流程、产品手册等），员工在日常办公中面临以下痛点：
- **知识分散**：制度散落在 OA、文档、邮件、IM 群中，新员工入职平均 2-3 周才能独立工作
- **检索低效**：找一份报销流程可能要在 5 个系统切换、问 3 个同事
- **理解门槛**：制度文件晦涩，HR/IT 每天被重复咨询 30+ 次相同问题
- **跨团队信息孤岛**：前端/后端/数据/运维知识共享困难
- **更新滞后**：文档更新到员工看到平均延迟 1-2 周，错过重要变更
- **个性化缺失**：同一份文档无法针对员工角色推送对应内容

### 1.3 项目目标
构建一个基于 **RAG + Agent + LangGraph** 三大 AI 引擎的企业级智能办公助手，实现：
- **四大模块化知识库**：规章制度 / 产品技术 / 行政服务 / 自由问答，每个模块独立 RAG
- **三种推理模式合一**：RAG（知识问答）+ Agent（工具调用）+ LangGraph（多步推理），同一界面自由切换
- **个人知识库自助管理**：用户可创建个人知识库，自由问答时多选使用
- **流式响应 + 工具调用可视化**：所有 AI 输出逐 token 实时推送，附件即时可下载
- **角色权限精细化**：管理员可创建共享知识库，普通用户仅可创建个人知识库
- **对话历史与收藏**：自动持久化所有对话，支持个人收藏

### 1.4 项目价值论证（对应实训要求）

本项目的选题严格遵循实训"自主命题"要求中的七大原则，以下逐一论证：

#### 1.4.1 真实需求导向 —— "为谁解决什么问题"

**目标用户**：科技企业内的技术团队（开发工程师、测试工程师、团队负责人、技术负责人）。

**核心痛点**：
- 企业技术文档（API 文档、架构设计、代码规范、运维手册等）分散在 Confluence、GitLab、共享盘等多个平台，员工查找某条技术规范平均需要 15-20 分钟
- 新员工入职后需要 2-4 周才能熟悉公司的技术规范和架构决策，期间效率低下
- 资深员工每天平均被问到 3-5 个重复性技术问题，打断深度工作
- 跨团队（前端/后端/数据/运维）知识共享困难，信息孤岛严重

**目前解决方案的不足**：当前企业多用 wiki + 全文搜索（如 Elasticsearch），但全文搜索无法理解语义（如"怎么处理 null 值"无法匹配到"空值处理规范"），且无法综合多个文档给出整合答案。

#### 1.4.2 商业前景明确 —— "谁来付费、市场规模如何"

**商业模式**：B2B SaaS 订阅制 + 私有部署授权。

- **目标客户**：50-500 人规模的技术驱动型企业（SaaS 公司、软件外包公司、金融科技公司等）
- **付费方**：企业 IT 部门或技术管理层
- **市场规模**：中国企业知识管理市场规模 2025 年预计 120 亿元，其中 AI 驱动的知识管理是增速最快的细分领域（年增长率 45%+）
- **定价策略**：SaaS 版 ¥999/月（50 用户），私有部署版 ¥49,900/年（含 1 年维保）
- **竞争优势**：相比 Dify/FastGPT 等通用平台，本系统聚焦科技企业场景，内置代码理解、技术文档解析等垂直能力

#### 1.4.3 AI 价值突出 —— "没有 AI 做不好"

本系统不是"传统信息系统 + AI 点缀"，AI 是系统的核心引擎：

| 传统方案 | 本系统 AI 方案 |
|----------|---------------|
| 全文关键词搜索 | 语义向量检索 + LLM 重排，理解查询意图 |
| 返回文档链接，用户自行阅读 | RAG 综合多文档生成整合答案，附来源标注 |
| 需要记住精确关键词才能找到内容 | 自然语言提问，支持多轮对话补充上下文 |
| 无法处理新出现的问题 | Agent 可联网搜索补充最新信息 |
| 仅能检索，不能执行任务 | LangGraph 智能体可多步推理、调用工具完成任务 |

**AI 不可替代性**：知识问答场景的核心挑战是"理解用户意图 → 检索相关信息 → 整合生成答案"，这三个环节分别依赖 LLM 的语义理解、向量检索、文本生成能力，传统方案无法胜任。

#### 1.4.4 数据可获得性

本系统的数据来源多样化且均可获取：

| 数据来源 | 类型 | 获取方式 |
|----------|------|----------|
| 企业内部文档 | PDF/Word/Markdown/HTML | 用户自行上传（系统不依赖外部数据） |
| 开源技术文档 | 网页 | SearXNG 联网搜索（聚合 Google/Bing/DuckDuckGo） |
| 代码仓库 | 代码片段 | 可扩展接入 GitLab/GitHub API |
| 测试示例数据 | Markdown | 系统内置示例文档（LLM 基础、RAG 介绍等） |

**降级策略**：即使联网搜索不可用，系统仍可基于本地知识库正常运行，核心 RAG 链路不依赖外部数据。

#### 1.4.5 实施可行性

**技术成熟度**：所有核心技术组件均有成熟的开源方案或低成本 API：

- ✅ FastAPI + Vue 3 + PostgreSQL：业界成熟的技术栈
- ✅ OpenSearch：AWS 开源的向量检索引擎，文档丰富
- ✅ Ollama + qwen3-embedding:0.6b：本地部署，零成本
- ✅ DeepSeek API：¥1/百万 Token，开发测试阶段几乎零成本
- ✅ SearXNG：开源元搜索引擎，免费自部署
- ✅ Redis：开源缓存数据库

**工期可行性**：4 人 × 13 天 = 416 人时，核心 RAG + Agent 功能已在 Sprint 1 完成，Sprint 2 补充高级功能，工期充裕。

#### 1.4.6 创新差异化

本项目相比市场上同类产品的差异化亮点：

1. **三种对话模式合一**：RAG（知识问答）+ Agent（工具调用）+ LangGraph（复杂任务编排），用户可根据需求切换模式，而非仅支持单一交互方式
2. **企业级权限模型**：全局知识库 + 个人知识库的隔离设计，管理员/普通用户角色分明，操作权限精确到知识库级别
3. **完整技术链路透明化**：从文档上传 → MinerU 解析 → 文本切片 → 向量化 → 检索 → Rerank → 生成，全链路可观测可调试
4. **本地 + 云端混合架构**：Embedding 模型本地部署保证数据隐私，LLM 推理使用 API 降低成本，兼顾安全与性能

#### 1.4.7 合规合法

- 用户数据存储在自有服务器，不上传第三方
- Embedding 模型（Ollama）本地运行，文档内容不出企业内网
- LLM 推理使用 DeepSeek API，仅传递检索到的文本片段（非完整文档）
- 系统不使用用户数据进行模型训练
- 不涉及金融、医疗、法律等强监管领域的合规风险

## 2. 需求调研

### 2.1 市场调研

#### 同类产品分析

| 产品 | 特点 | 不足 |
|------|------|------|
| **Dify** | 开源 LLM 应用开发平台，支持 RAG | 部署复杂，企业定制性弱 |
| **FastGPT** | 知识库问答系统，可视化编排 | 中文支持好，但权限管理简单 |
| **ChatGLM** | 智谱 AI 的企业知识库方案 | 依赖云端 API，数据隐私顾虑 |
| **LlamaIndex** | RAG 框架，灵活可定制 | 需要较多开发工作 |

#### 调研结论
1. **开源方案**：多数开源 RAG 方案侧重于技术演示，缺乏完整的企业级功能（权限管理、多租户、审计日志）
2. **商业方案**：商业产品功能完善但成本高，且数据需要上传到第三方服务器
3. **技术选型**：采用 FastAPI + Vue 3 + PostgreSQL + OpenSearch 的技术栈，兼顾开发效率和系统性能

### 2.2 用户需求调研

#### 目标用户群体
| 角色 | 核心需求 | 使用场景 |
|------|----------|----------|
| **技术负责人** | 全局知识管理、权限控制 | 审批知识库创建、查看使用统计 |
| **团队负责人** | 团队知识沉淀、文档管理 | 上传团队文档、管理团队成员 |
| **开发工程师** | 快速查找技术方案、API 文档 | 日常开发中查询技术规范 |
| **测试工程师** | 查询测试规范、历史 Bug 解决方案 | 编写测试用例时参考历史经验 |

#### 核心功能需求
1. **知识库管理**：支持创建多个知识库，按团队/项目隔离
2. **文档解析**：自动解析 PDF、Word、Markdown 等格式，提取文本并生成向量
3. **智能问答**：基于本地知识库检索 + LLM 生成，支持联网搜索补充
4. **权限管理**：基于 RBAC 的权限控制，不同角色有不同操作权限
5. **对话历史**：保存对话记录，支持多轮对话上下文

### 2.3 技术调研

#### 2.3.1 为什么选择 RAG 架构？

**问题**：企业知识问答需要基于私有数据，直接用 LLM 无法获取企业内部文档内容，且存在数据泄露风险。

**调研方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **微调 LLM** | 回答质量高 | 训练成本高、更新困难、需要大量 GPU | 领域固定、数据量小 |
| **RAG（检索增强生成）** | 知识可动态更新、成本低、可溯源 | 检索质量影响回答质量 | 知识频繁更新、需要引用来源 |
| **纯 LLM + Prompt** | 实现简单 | 无法获取私有知识、幻觉严重 | 通用问答 |

**结论**：选择 RAG 架构，因为企业知识需要频繁更新，且需要标注回答来源，便于用户验证。

#### 2.3.2 向量数据库选型

| 数据库 | 优点 | 缺点 | 部署复杂度 |
|--------|------|------|------------|
| **OpenSearch** | 支持混合检索、权限控制、大规模部署 | 内存占用较高 | 中等 |
| **Milvus** | 专为向量设计、性能优秀 | 混合检索支持弱 | 高 |
| **pgvector** | 与 PostgreSQL 集成、部署简单 | 大规模性能不如专用向量库 | 低 |
| **Chroma** | 轻量级、开发友好 | 生产环境稳定性不足 | 低 |

**结论**：选择 OpenSearch，因为企业场景需要混合检索（向量 + 关键词）、权限过滤，且 OpenSearch 生态成熟，支持大规模部署。

#### 2.3.3 Embedding 模型选型

| 模型 | 维度 | 中文支持 | 部署方式 | 性能 |
|------|------|----------|----------|------|
| **qwen3-embedding:0.6b** | 2560 | 优秀 | 本地 Ollama | 高 |
| **nomic-embed-text** | 768 | 一般 | 本地 Ollama | 中 |
| **text-embedding-3-small** | 1536 | 良好 | OpenAI API | 高 |
| **bge-large-zh** | 1024 | 优秀 | 本地部署 | 中 |

**结论**：选择 qwen3-embedding:0.6b，因为中文支持优秀，本地部署保证数据隐私，且 2560 维向量能提供更精细的语义表示。

#### 2.3.4 LLM 选型

| 模型 | 中文能力 | 成本 | 响应速度 | 部署方式 |
|------|----------|------|----------|----------|
| **DeepSeek** | 优秀 | 低（1元/百万Token） | 快 | API |
| **GPT-4** | 良好 | 高（$30/百万Token） | 中 | API |
| **Qwen-Max** | 优秀 | 中 | 快 | API |
| **本地 Llama3** | 一般 | 免费（需 GPU） | 慢 | 本地 |

**结论**：选择 DeepSeek API，因为中文能力优秀，成本极低，且响应速度快，适合企业级应用。

#### 2.3.5 联网搜索方案

| 方案 | 优点 | 缺点 | 隐私保护 |
|------|------|------|----------|
| **SearXNG** | 开源、聚合多引擎、隐私保护 | 需要自建服务 | 优秀 |
| **Bing API** | 结果质量高 | 收费、数据外传 | 差 |
| **Google API** | 结果最全面 | 收费、国内访问困难 | 差 |

**结论**：选择 SearXNG，因为开源免费，支持聚合多个搜索引擎，且保护用户隐私，适合企业内网部署。

### 2.4 问题解决

#### 2.4.1 文档解析质量问题

**问题**：PDF、Word 等格式解析后文本混乱，表格、图片内容丢失。

**解决方案**：
1. 使用 MinerU API 进行专业文档解析，支持保留文档结构
2. 对表格进行特殊处理，转换为 Markdown 格式
3. 图片内容使用 OCR 提取文字
4. 提供手动调整切片大小的配置，适应不同文档

#### 2.4.2 检索准确率问题

**问题**：向量检索返回的结果相关性不高，包含大量无关内容。

**解决方案**：
1. **查询重述**：在多轮对话中，使用 LLM 补全省略的上下文，生成完整查询
2. **Rerank 精排**：使用 LLM 对检索结果重新打分，保留最相关的 Top-3
3. **混合检索**：结合向量检索和关键词检索，提高召回率
4. **元数据过滤**：支持按知识库、文档类型、时间范围过滤

#### 2.4.3 多轮对话上下文丢失

**问题**：用户在多轮对话中，AI 无法理解省略的指代内容。

**解决方案**：
1. 使用 Redis 存储对话历史，支持多轮上下文
2. 在每次查询前，使用 LLM 进行查询重述，补全省略内容
3. 限制历史轮数（默认 10 轮），避免上下文过长影响性能

#### 2.4.4 权限控制问题

**问题**：不同用户应该看到不同的知识库，如何在不泄露数据的前提下实现权限控制？

**解决方案**：
1. **知识库隔离**：每个知识库对应独立的 OpenSearch 索引
2. **检索时过滤**：在检索时根据用户权限过滤可访问的知识库
3. **前端菜单控制**：根据用户角色动态显示菜单项
4. **后端接口校验**：所有接口都进行权限校验，防止越权访问

#### 2.4.5 系统性能优化

**问题**：文档解析和向量生成耗时较长，影响用户体验。

**解决方案**：
1. **异步处理**：文档上传后立即返回，后台异步解析
2. **进度反馈**：通过轮询显示文档解析进度
3. **批量处理**：向量生成使用批量接口，减少 API 调用次数
4. **缓存策略**：热门问题答案缓存到 Redis，减少重复计算

## 3. 功能需求

### 3.1 用户认证与权限管理

#### 3.1.1 用户登录
- 支持用户名/密码登录
- JWT Token 认证，Token 有效期 24 小时
- 登录失败提示，连续失败 5 次锁定账号 30 分钟

#### 3.1.2 角色权限
| 角色 | 权限说明 |
|------|----------|
| **管理员** | 所有权限，可管理所有知识库和用户，创建全局知识库 |
| **普通用户** | 查询知识库，上传文档，使用对话机器人，创建个人知识库 |

#### 3.1.3 预置账号
系统预置以下测试账号：
- **管理员账号**：admin（密码：admin123）
- **测试用户**：test（密码：test123）

#### 3.1.4 用户管理
- 管理员可创建、编辑、删除用户
- 新注册用户默认为普通用户
- 支持修改个人信息（姓名、邮箱、部门、头像、密码）
- 支持账户注销（管理员和普通用户均可注销自己的账户）

### 3.2 知识库管理

#### 3.2.1 知识库类型
系统支持两种知识库类型：
- **全局知识库**：管理员创建，所有用户可见，用于存储企业级公共知识
- **个人知识库**：普通用户创建，仅创建者可见，用于存储个人学习资料

#### 3.2.2 知识库 CRUD
- 创建知识库：名称、描述、所属团队、负责人、Embedding 模型、是否个人知识库
- 编辑知识库：修改基本信息（管理员可编辑所有，普通用户仅可编辑自己的个人知识库）
- 删除知识库：软删除，保留历史数据（权限同编辑）
- 知识库列表：支持按名称/团队筛选，自动根据用户权限过滤

#### 3.2.3 多知识库隔离
- 每个知识库对应独立的 OpenSearch 索引
- 检索时支持指定知识库范围
- 管理员可跨知识库检索，普通用户仅可检索全局知识库和自己的个人知识库

#### 3.2.4 四大业务模块
系统按业务场景预置四大模块，每个模块挂载一个或多个知识库：

| 模块 | 图标 | 应用场景 | 默认知识库 |
|------|------|---------|-----------|
| **规章制度** | 📋 | 报销、考勤、请假、绩效 | `policy`（制度文档库） |
| **产品技术** | 💻 | API 文档、架构、运维 | `tech`（技术手册库） |
| **行政服务** | 🏢 | 场地、IT 设备、福利 | `admin`（行政手册库） |
| **自由问答** | 💬 | 任何开放性问题 | `general`（通用库）+ 用户个人库 |

**模块化设计要点**：
- 每个模块的初始知识库**锁定不可删除**（防止误删破坏系统）
- 用户切换模块时，RAG 检索自动限定在该模块的知识库范围内
- 自由问答模块可多选知识库（通用库 + 自己的个人库）
- 管理员可对前三个模块补充共享知识库，普通用户不能

### 3.3 文档管理

#### 3.3.1 文档上传
- 支持格式：PDF、DOCX、PPTX、XLSX、MD、TXT、HTML
- 拖拽上传，显示上传进度
- 文件大小限制：默认 50MB

#### 3.3.2 文档解析
- 使用 MinerU API 解析文档，提取文本和结构
- 自动文本切片（chunk_size=500, chunk_overlap=100）
- 使用 Ollama 生成文本向量（qwen3-embedding:0.6b）
- 向量存储到 OpenSearch

#### 3.3.3 文档管理
- 文档列表：显示文件名、大小、切片数、状态
- 文档预览：查看解析后的文本内容
- 文档删除：同步删除向量数据

### 3.4 对话机器人

#### 3.4.1 RAG 完整链路
```
用户提问 → 查询重述 → 向量检索 → Rerank 精排 → [可选联网搜索] → 构建上下文 → LLM 生成 → 流式输出
```

1. **查询重述**：多轮对话中补全省略/指代，提高检索准确率
2. **向量检索**：从 OpenSearch 检索 Top-K 相关文档片段
3. **Rerank 精排**：使用 LLM 对检索结果重新打分，保留最相关的 3 条
4. **联网搜索**：用户可选择开启，使用 SearXNG 搜索互联网补充信息
5. **答案生成**：基于本地知识库优先，联网结果作为补充

#### 3.4.2 智能体（Agent）开发

系统具备工具调用（Function Calling）能力，大模型可根据用户意图主动操作数据库、调用外部 API、执行计算任务。

**可用工具集**：

| 工具名称 | 功能描述 | 输入参数 | 输出格式 |
|---------|---------|---------|---------|
| `search_knowledge_base` | 搜索本地知识库 | query: 查询词, top_k: 返回数量 | 文档片段列表 |
| `web_search` | 搜索互联网 | query: 查询词 | 网页结果列表 |
| `calculate` | 执行数学计算 | expression: 数学表达式 | 计算结果 |
| `get_system_info` | 获取系统信息 | 无 | 系统状态信息 |

**Agent工作流程**：
```
用户提问 → LLM分析意图 → 决定调用工具 → 执行工具 → 获取结果 → 生成最终回答
```

#### 3.4.3 高级智能体编排（LangGraph）

采用图编排框架构建具备状态管理与长短期记忆能力的高级智能体，支持多步推理、自我修正、异常处理。

**LangGraph智能体架构**：
```
┌─────────────┐
│  任务分析   │ ← 分析用户意图，决定是否需要工具
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌─────────────┐
│  工具执行   │ ←── │  条件判断   │
└──────┬──────┘     └─────────────┘
       │
       ▼
┌─────────────┐
│  答案生成   │ ← 基于工具结果生成最终答案
└─────────────┘
```

**状态管理**：
- `messages`: 对话消息历史
- `next_action`: 下一步动作（use_tool/answer/end）
- `tool_calls`: 工具调用列表
- `tool_results`: 工具执行结果
- `final_answer`: 最终答案
- `iteration`: 迭代次数（防止无限循环）

**三种对话模式**：
1. **RAG模式**：标准检索增强生成，适合知识问答
2. **Agent模式**：工具调用，适合需要执行操作的任务
3. **LangGraph模式**：复杂任务编排，适合多步骤推理

#### 3.4.4 对话功能
- **流式输出（SSE）**：逐 token 实时显示 AI 回答，三种模式（RAG/Agent/LangGraph）均支持流式
- **多轮对话**：通过 Redis 保存对话历史，支持上下文
- **会话管理**：创建新对话、切换历史对话、删除对话
- **跨页面持久化**：会话 ID 与当前模块写入 localStorage，刷新/跳转页面不丢失
- **对话按模块归类**：每条对话自动关联到发起时的模块，历史列表显示模块小徽标
- **来源标注**：回答中标注引用来源，支持跳转原文
- **模式切换**：用户可选择 RAG/Agent/LangGraph 模式
- **个人收藏**：用户可收藏重要对话，侧边栏"我的收藏"统一管理
- **附件下载**：Agent 生成的 Word/PDF 文档即时渲染为可下载卡片
- **幻觉兜底**：检测 LLM 假装调工具但未实际调用的情况，自动补救生成真实文件

#### 3.4.5 检索配置
- Top-K：默认 5 条，可调整 1-20
- 相似度阈值：默认 0.5，可调整 0-1
- 联网搜索开关：用户可手动切换

### 3.5 数据洞察

#### 3.5.1 对话日志
- 记录每次问答的详细信息（用户、问题、回答、来源、时间）
- 支持按用户/时间/关键词筛选
- 管理员可查看所有日志，普通用户仅查看自己的日志

#### 3.5.2 运营统计
- 总咨询量、活跃用户数、平均响应时长
- 热门问题排行榜
- 知识库使用统计

### 3.6 系统配置

#### 3.6.1 可配置项
- 文档解析参数：上传大小限制、切片大小、切片重叠
- 检索参数：Top-K、相似度阈值、检索超时
- 对话参数：最大上下文轮数、最大回答长度、回答超时
- LLM 参数：模型温度、最大 Token 数

## 10. 风险与应对
